## Project Overview

**Capture Forge** — a browser extension (Chrome first, Firefox planned) for screen recording and video editing, written in Rust and compiled to WebAssembly via the [oxichrome](https://crates.io/crates/oxichrome) framework.

The product follows a phased roadmap per `docs/product-brief.md` (finalized) and `docs/prd.md`:
1. **Recorder Core** (P0, V0.1) — screen/tab capture + mic + pause/resume/stop + WebM export + OPFS storage + crash recovery
2. **Editor + Overlay** (P1, V0.5) — non-destructive trim/mute/crop, toolbar, canvas annotations, camera PiP, Firefox support, MP4/GIF export
3. **AI & Semantic** (P2, V2.0+) — local STT (sherpa-onnx/WASM), optional LLM, DOM capture, Audience Lenses

**Key decisions** (per product brief): MIT license, GitHub + Chrome Web Store distribution, Firefox in P1 not P0, local-first/privacy-first positioning.

**Current state:** Scaffold. The extension registers a service worker that logs startup and install events. Core recording engine not yet implemented.

## Prerequisites

```bash
rustup target add wasm32-unknown-unknown
cargo install wasm-pack
# Optional: needed only when regenerating manifest/JS shims
cargo install cargo-oxichrome
```

## Build Commands

```bash
# Fast compile-time validation (no wasm target needed)
cargo check

# Recompile Rust → WASM (day-to-day workflow)
wasm-pack build --target web

# Full oxichrome pipeline — regenerates manifest.json, background.js, and JS shims
cargo oxichrome build          # Debug
cargo oxichrome build --release  # Optimized
cargo oxichrome clean          # Remove dist/
```

### Build output

```
dist/chromium/
  manifest.json           # Manifest V3 (edit when permissions/entry points change)
  background.js           # ES module service worker
  wasm/
    capture_forge.js      # wasm-bindgen JS glue (regenerated on every wasm-pack build)
    capture_forge_bg.wasm # WebAssembly binary (~361 KiB debug)
    capture_forge.d.ts    # TypeScript declarations
```

The checked-in manifest and background.js were generated by oxichrome. Edit them directly when adding permissions or entry points.

### Adding a popup or options page

Add `#[oxichrome::popup]` or `#[oxichrome::options_page]` to a Leptos component function in `src/lib.rs`, then run `cargo oxichrome build` (not `wasm-pack` alone — the CLI generates the HTML/JS shims). **Leptos is not yet in `Cargo.toml`** — add it when implementing a popup.

## Development Workflow

1. Edit Rust code in `src/lib.rs` or add new modules under `src/`
2. Run `wasm-pack build --target web`
3. Load `dist/chromium/` as an unpacked extension at `chrome://extensions/` (Developer mode on)
4. Inspect the service worker via Extensions → Capture Forge → Service Worker → Console

When changing permissions, extension name, or version, update **both** `src/lib.rs` and `dist/chromium/manifest.json`.

## Architecture

### Current crate layout

```
src/lib.rs         # Single Rust entry point — Extension struct, background handler, install handler
Cargo.toml         # cdylib crate (targets wasm32-unknown-unknown)
Oxichrome.md       # Local reference copy of Oxichrome docs (version 0.2)
```

### Data flow

```
src/lib.rs (Rust)
    │ oxichrome proc macros + wasm-bindgen
    ▼
dist/chromium/wasm/capture_forge.js + capture_forge_bg.wasm
    │ background.js imports and calls __oxichrome_bg_start()
    ▼
Chrome service worker (Manifest V3)
```

### Extension lifecycle (three proc-macro attributes)

1. **`#[oxichrome::extension(...)]`** — declares the extension struct (name, version, permissions), generates manifest metadata + JS↔WASM binding
2. **`#[oxichrome::background]`** — async function running when the service worker starts
3. **`#[oxichrome::on(runtime::on_installed)]`** — handler for chrome.runtime.onInstalled

### Implementation roadmap (from technical research)

The P0 capture engine uses a `tabCapture → offscreen document → MediaRecorder → OPFS` chain. Key architectural constraints:

- **Service worker** cannot access DOM/media APIs in MV3 — an **offscreen document** is mandatory for recording
- **SW idle timeout** is ~30s after last event; single operation limit is ~5 min → resilience via design-for-resurrection (WAL, heartbeat, offscreen outlives SW)
- **OPFS** with `FileSystemSyncAccessHandle` provides crash-safe chunked writes

See `docs/architect.md` for the full module structure and `_bmad-output/planning-artifacts/research/` for detailed feasibility analysis and implementation phasing.

### Planned module structure (per docs/architect.md)

The `src/` directory is expected to grow into:

```
src/lib.rs           → #[extension] entry
src/background/       → Service worker (listeners, messaging)
src/recorder/         → RecordingSession state machine
src/storage/          → Storage trait + OPFS + IndexedDB fallback
src/export/           → WebM concatenation
src/editor/           → Non-destructive editor (P1)
src/content_script/   → Overlay + annotations (P1)
src/camera_page/, src/region_page/, src/popup/, src/setup/
src/ai/               → P2 optional (STT, LLM, DOM capture)
```

JS shims bridge where web-sys doesn't expose Chrome APIs:
- `js/chrome_shim.js` — tabCapture, offscreen document API
- `js/ffmpeg.js` — FFmpeg WASM (P1)
- `js/mediapipe.js` — MediaPipe (optional, P2)
